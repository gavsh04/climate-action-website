Feedback & Programme Directory Page

Technical Description

The feedback.html page is structured using the shared team template, linking both style.css (global) and feedback.css (page-specific). The page is divided into three main functional sections, each wrapped in semantic HTML elements.

Programme Directory Section: A scrollable card grid was created using a <div class="programme-scroll"> container with overflow-y: auto and a fixed max-height in CSS. Each programme is an <article> element containing an image, heading, paragraph description, and a styled anchor button. The grid layout uses CSS display: grid with grid-template-columns: repeat(auto-fit, minmax(220px, 1fr)), making it fully responsive. A styled <select> dropdown provides a sorting UI (visual only, no functional filtering required by the spec).

Ratings & Reviews Section: A large numeric rating (8.7) is displayed using a large font-size span styled in feedback.css. Three review cards are arranged using display: flex; flex-wrap: wrap. Each card has a coloured left border (border-left: 4px solid #00c71c) to create visual accent without heavy design overhead.

Feedback Form Section: The form is built with two <fieldset> groups — "Personal Details" and "Feedback Details" — each containing a <legend> for accessibility grouping. Fields include type="text", type="email", type="tel", type="date", type="datetime-local", a <textarea> with maxlength="300", a <select> dropdown, and a <datalist> for searchable suggestions. A live character counter (<span id="charCount">) is updated by JavaScript as the user types. Form validation is handled by submitFeedback() in nadith script.js, which checks required fields, shows .error-msg spans near invalid inputs, and on success reveals a #confirmation-msg div via DOM manipulation.

Key CSS Properties: overflow-y: auto for scrollable section; display: grid / flex for responsive card layouts; CSS transitions (transform: translateY / scale) for hover effects on programme cards and submit button; ::-webkit-scrollbar custom styling for the scroll track; :focus pseudo-class highlighting for form fields; display: none toggled to show error messages and confirmation via JavaScript.

Accessibility

All programme card images include descriptive alt text explaining the scene, not just the file name. Each form input has an explicit <label for="..."> matching its id, ensuring screen readers announce the field purpose correctly. The two <fieldset> / <legend> groups allow assistive technology to present the form in logical chunks. The confirmation message uses role="alert" and aria-live="polite" so screen readers announce it without requiring focus. The sort dropdown carries an aria-label attribute. Colour contrast between the green palette (#2E7D32) and white text was verified to meet WCAG AA standards.

Team Page

Technical Description

The team.html page uses the shared template with style.css as the global stylesheet and team.css as the page-specific external CSS. No embedded <style> block was needed because the page-specific rules are entirely encapsulated in team.css.

Hero Banner: A full-width hero is created using a <section class="team-hero"> with a CSS background combining a dark linear gradient overlay (rgba(0,0,0,0.52)) over a high-resolution Unsplash image. This ensures text readability regardless of the image brightness.

Team Grid: Four member blocks are arranged inside a .team-grid container using display: grid; grid-template-columns: repeat (auto-fill, minmax(240px, 1fr)). This creates a fluid responsive layout that naturally reflows from 4 columns on large screens to 1 column on mobile.

Interactive Hover-Expand Cards: Each .team-member card uses a CSS max-height: 0; overflow: hidden; transition: max-height 0.5s ease technique on the .member-details panel. On :hover and :focus, the max-height is expanded to 300px, smoothly revealing the contribution summary, role, and list of pages. The thumbnail image simultaneously fades to opacity: 0.6, hinting to users that more content is available below. A transform: translateY(-10px) scale(1.02) on hover creates a subtle lift effect.

Keyboard Accessibility (tabindex): Each .team-member div carries tabindex="0" and role="button" so keyboard users can Tab into each card and trigger the same hover state via :focus. This means the expand behaviour works identically with the keyboard as with the mouse — no JavaScript required for the interaction.

Team Statement: A closing <section class="team-statement"> provides a shared mission paragraph using semantic HTML without extra markup complexity.

Key CSS Properties: grid-template-columns: repeat(auto-fill, minmax(...)) for responsive grid; max-height transition for expand effect; transform and box-shadow transitions for hover lift; object-fit: cover; object-position: top center for portrait photos; outline: 3px solid #2E7D32 on focus for visible focus ring.

Accessibility

All team member photos include alt attributes describing the person, their student number, and their responsibility. Each interactive card has tabindex="0" for keyboard navigation and role="button" with a descriptive aria-label (e.g. "Nadith – Student 3, Feedback and Team pages. Hover for more details."), giving screen reader users context before they interact. The :focus state applies the same visible green outline as the hover state, meeting WCAG 2.1 Focus Visible (2.4.7) criteria. The CSS-only expand requires no scripting, reducing complexity and failure points for assistive technology.

Content Page – Climate Policy Changes

Technical Description

The content_ST3.html page is the individual content page covering Climate Policy Changes. It links style.css (global) and content.css (page-specific external stylesheet). The page uses the id="top" attribute on <body> as the anchor target for the "Back to Top" button.

Page Structure: The page is divided into five content sections — Why Policy Matters, The Paris Agreement, UK Climate Policy, Challenges, and What Students Can Do — each identified with a unique id attribute. This enables the Table of Contents (<nav class="toc">) at the top of <main> to use anchor links (href="#why", href="#paris", etc.) for in-page navigation.

Table of Contents: Implemented as a <nav> with aria-label="Page contents" and an ordered list (<ol>) of anchor links. The .toc class in content.css applies a green left border (border-left: 4px solid #4CAF50), a light green background, and padding to visually distinguish it from body text.

Images: Each section includes a full-width image styled with width: 100%; height: 260px; object-fit: cover; border-radius: 8px. All images are sourced from Unsplash (royalty-free) and include descriptive alt text and a <p class="caption"> below each image crediting the source.

Info Boxes: Highlighted key facts are wrapped in <div class="info-box"> elements styled with a left green border and italic font to draw attention to critical statistics.

"Go to Top" Button: An <a href="#top" id="go-top"> element is positioned with position: fixed; bottom: 28px; right: 28px and styled as a circular green button, always visible as the user scrolls. No JavaScript is used — this is a pure CSS/HTML implementation as required.

Key CSS Properties (content.css): .content-hero — gradient overlay + background image for the hero; .page-content — max-width: 800px; margin: 0 auto for readable line length; .toc — accent border and background; #go-top — fixed circular button; .info-box — highlighted callout block; .caption — small italic image credit.

Accessibility

Heading hierarchy is maintained throughout: one <h1> in the hero section followed by <h2> for each content section. All four images include specific alt text describing the depicted scene. The Table of Contents <nav> carries an aria-label so screen readers can distinguish it from the main navigation bar. The "Go to Top" button uses an aria-label="Back to top" to provide a meaningful label for its upward arrow symbol. Colour contrast between #2E7D32 headings and the white background was checked to meet WCAG AA. Lists use <ul> / <li> elements correctly rather than styled paragraphs, preserving list semantics for screen readers.

Challenges and Lessons Learned

CSS-only hover-expand on team cards: My initial approach used JavaScript click events to toggle a class, but the specification required the Team Page interactivity to be CSS-based. I refactored to use the max-height transition technique — setting max-height: 0 by default and max-height: 300px on :hover / :focus. The key lesson was that height: auto cannot be transitioned, so max-height with a reasonable upper bound is the standard workaround. I also learned that tabindex="0" combined with :focus pseudo-class is the correct approach for making non-anchor, non-button elements keyboard accessible.

JavaScript form validation — error message placement: Early versions of the feedback form showed a single alert dialog for all errors, which was disruptive. I refactored to use inline <span class="error-msg"> elements placed directly below each input, shown and hidden by toggling display: none / block via DOM manipulation. This required careful co-ordination between feedback.css (styling the spans) and nadith script.js (controlling visibility). The lesson was to design HTML structure and JavaScript interaction together from the start rather than retrofitting.

Scrollable section height: The programme directory initially collapsed to zero height on some screen sizes because the parent container had no explicit height and overflow-y: auto requires the container to have a bounded height. Setting max-height: 520px on .programme-scroll resolved this. I also added custom scrollbar styling (::-webkit-scrollbar) to maintain the green visual identity inside the scrollable container.

Image sourcing and referencing: Using Unsplash URLs directly in src attributes was convenient for development but risks broken images if URLs change. This reinforced the importance of downloading royalty-free assets locally and referencing them with relative paths, which I have documented in the References section below.

Compliance with JANET Regulations

All pages I implemented comply with JANET regulations governing web page publication in the following ways:

Content appropriateness: All written content on feedback.html, team.html, and content_ST3.html is original, factual, and appropriate for a university academic context. No offensive, defamatory, or discriminatory material is present.

Copyright and external resources: All images are sourced from Unsplash (royalty-free, free for commercial and personal use) or are photographs of team members used with their consent. All external image sources are referenced in the References section of this Page Editor. No copyrighted third-party content has been reproduced without permission.

Privacy: The feedback form does not transmit data to any server. JavaScript validation runs entirely client-side, and no personal data entered by users is stored or processed beyond the current browser session. This is made clear by the form's confirmation message.

Accessibility: Semantic HTML, ARIA attributes, and WCAG-compliant colour contrast are used throughout. This aligns with the University of Westminster's commitment to inclusive digital practice as required under JANET acceptable use policies.

Honest representation: Team member information on team.html represents actual contributions accurately. No misleading content is published.
