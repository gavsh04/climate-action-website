Reflection on the Validation Process

Validating all three pages using the W3C Markup Validation Service was a valuable and instructive part of the development process. The validator acts as a strict, standards-based referee — catching issues that browsers silently ignore but that can cause inconsistent behaviour across platforms.

Errors encountered and resolved: During development of the content page, I had accidentally left a stray closing </p> tag after a <div> element, which the validator flagged immediately. This was a simple fix but one I would not have spotted by just looking at the rendered page, since browsers auto-correct such issues silently. I also discovered that I had used an alt attribute on a <div> (copied from an earlier draft) which is not a valid attribute for that element — this was removed and moved to the correct <img> element.

SVG validation: Validating the sitemap page required extra care because the SVG is embedded inline within the HTML document. I had initially included xmlns="http://www.w3.org/2000/svg" on the <svg> element, which the validator indicated was redundant in HTML5 (though not an error). I left it in for compatibility clarity.

Key lesson: Running validation frequently during development — rather than only at the end — helps catch errors early before they compound. I now treat the W3C validator as a routine check alongside browser testing, not a final step.
